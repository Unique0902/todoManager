✅ 정확하게 잘 이해한 부분들
전체 컨셉 요약: 목표/프로젝트/루틴 등 다양한 단위를 트리로 관리한다는 개념 → 정확함.

트리 구조 규칙: 각 타입별 자식 가능 여부 및 마일스톤 특성 부여 규칙 → 대부분 정확.

루틴의 관리 방식: 수행률, 난이도 조정, 감정일기 루틴 등 상세한 설정 반영 → 잘 이해함.

트리뷰(목표트리) 설계 방향: 피그마 스타일 인터랙션, 자식 추가만 가능 → 정확함.

홈 화면 기능 구성: 지금 해야 할 일 / 오늘의 할 일 / 오늘의 성과 / 피드백 흐름 → 너가 말한 의도대로 잘 정리됨.

감정일기 구조 및 역할: 하루 여러 개 가능, 루틴과 연결, 피드백 흐름에 포함 → 정확.

캘린더 시각화: 할일 vs 프로젝트/마일스톤 구분, 줄띠형 표시 등 → 네 디테일 그대로 잘 반영됨.

아이디어/버킷리스트 별도 페이지로 분리, 트리와 무관 → 👍

⚠️ 확인하거나 수정할 필요가 있는 부분
1. ❗️“마일스톤 특성 노드”라는 용어 사용
🔴 문제점: milestone 특성 노드라는 표현이 마치 type처럼 보임 → 혼동 유발.

✅ 수정 제안: "마일스톤 특성을 가진 노드"로 표기 / milestone_group의 자식은 is_milestone: true 자동 부여.

2. ⚠️ 프로젝트와 목표의 차이점 강조 부족
커서 요약에서 goal과 project의 실질적 차이에 대한 서술이 조금 약함.

✅ 보완 포인트:

goal: 방향성/추상적, 기간 없이도 존재 가능

project: 구체적 실행 단위, 기간 필수, 여러 할일/루틴/마일스톤 포함 가능

3. 🔸트리뷰 내 진행률 표시 규칙
"루틴을 자식으로 갖지 않는 경우만 진행률 표시" → 맞지만 추가로 명시하면 좋음:

진행률은 자식 노드 중 task 또는 project 등 체크 가능한 항목들 기준으로 계산됨

자식이 모두 완료되어야 100%, 루틴 포함시 진행률 제외

4. 🔹루틴의 “수행률” 표현이 누락된 곳 있음
커서 요약 일부에서는 루틴 성공률이라고 되어 있음 → ✅ 수행률로 통일 필요

5. 📌 감정일기 데이터 구조 내 routine_id 필드
현재는 감정일기가 루틴과 1:1 대응처럼 보이는데, 사실은 루틴과 무관하게도 작성 가능

✅ 수정 제안:

감정일기 모델: date, content, created_at, (optional) linked_routine_id

그리고 "오늘 피드백"에서는 감정일기 작성을 루틴 유도 흐름과 별도로 처리

6. 🔸분석/통계 페이지 기능
“진행률”은 project/milestone_group에만 해당되므로, 통계에서도 이들 중심으로 시각화해야 함.

루틴의 경우 "수행률 + 연속 성공일 수 + 난이도 조정 횟수"가 메인 지표임을 더 명확히 하면 좋음

✅ 추가 제안
트리뷰 필터: 루틴/할일 숨기기 기능 → 시야가 복잡해질 경우에 대비

노드 고정 기능: 자주 사용하는 goal이나 project를 상단 고정 가능하게 (즐겨찾기처럼)
